### 1.2 디자인 토큰 평가

**양호한 점:**
- Tailwind CSS 색상 팔레트와 일관성 있는 Slate/Indigo 계열 사용
- 그라디언트가 브랜드 식별에 효과적 (Primary CTA의 Indigo->Violet)
- 그림자 3단계 계층이 명확 (sm/md/lg)
- `--radius-xl`~`--radius-sm` 4단계가 일관됨

**문제점:**

| ID | 문제 | 심각도 |
|----|------|--------|
| DT-1 | `--gradient-warm`, `--gradient-cool` 정의만 있고 미사용 (Dead Token) | P3 |
| DT-2 | `--ink-muted` (#94a3b8)의 `--bg` (#f8fafc) 대비 대비율이 약 **3.0:1**로 WCAG AA 미달 (4.5:1 필요) | P1 |
| DT-3 | `--warning` (#f59e0b)이 흰색 배경에서 텍스트로 직접 사용되면 대비율 부족 | P2 |
| DT-4 | 간격(spacing) 토큰이 없음. 코드에서 `12px`, `16px`, `20px`, `24px`, `40px` 등 매직넘버 직접 사용 | P2 |
| DT-5 | 버튼 `border-radius: 999px` (pill) vs 카드 `--radius-xl: 20px` -- 동일 인터페이스에 2가지 모서리 체계 혼재 | P3 |

## 2. 페이지별 UX 분석

### 2.1 게스트 랜딩 (비로그인 홈)

**5초 테스트:**
- 첫 인상: 깔끔하고 목적이 분명한 히어로 섹션. "출퇴근을 책임지는 앱"이 즉시 눈에 들어옴
- CTA 가시성: "무료로 시작하기" 버튼이 충분히 눈에 띔 (gradient + shadow)
- 정보 구조: 3-step feature cards가 가치 제안을 빠르게 전달

**양호한 점:**
- Nielsen H8 (Aesthetic & Minimalist): 불필요한 정보 없이 핵심만 전달
- Gestalt 연속성: 1-2-3 넘버링으로 플로우 방향 암시
- CTA 계층: 상단 "시작하기" (secondary) + 중앙 "무료로 시작하기" (primary) 이중 CTA

**발견된 이슈:**

| ID | 우선순위 | 휴리스틱/원칙 | 위치 | 현재 | 제안 | 근거 |
|----|---------|-------------|------|------|------|------|
| G-1 | P2 | Fitts's Law | 하단 네비게이션 | 비로그인에도 홈/경로/알림/설정 4탭 모두 표시. 클릭 시 각 페이지에서 로그인 유도 | 비로그인 시 하단 탭을 숨기거나 축소 표시 | 사용 불가능한 기능에 대한 탐색 비용 발생. Hick's Law 위반 (불필요한 선택지) |
| G-2 | P2 | Gestalt: Figure-Ground | Feature cards | 카드와 배경의 경계가 불분명 (1px border만 있음). 모바일에서 카드 구분 어려움 | 카드에 `--shadow-sm` 추가 또는 배경색 차이 강화 | 카드 간 시각적 분리 부족 |
| G-3 | P3 | Peak-End Rule | 페이지 하단 푸터 | "출퇴근 메이트" 텍스트만 있는 빈약한 푸터 | 하단 네비게이션 위에 불필요한 푸터 제거, 또는 소셜 프루프(사용자 수 등) 추가 | 마지막 인상이 빈약 |
| G-4 | P1 | Nielsen H2 (Real World Match) | 알림 페이지 (비로그인) | "! 로그인 후 알림을 설정할 수 있어요." 라는 warning 배너 하나 + 빈 페이지. 위저드 단계 표시("단계")가 깨져서 보임 | 알림 페이지도 경로/설정처럼 중앙 정렬 빈 상태 디자인 사용. 위저드 표시를 비로그인 시 완전 숨김 | 페이지 간 비로그인 처리가 3가지 패턴으로 불일치 (홈=전용 랜딩, 경로/설정=빈 상태+CTA, 알림=warning 배너+깨진 위저드) |

### 2.2 홈 페이지 (로그인 후)

**5초 테스트 (코드 기반 분석):**
- 첫 인상: 인사말 -> 날씨 카드 -> 준비물 체크리스트 -> 출퇴근 카드 -> 알림 -> 통계 순서의 수직 플로우
- CTA 가시성: "출발하기" 버튼이 충분히 크고 눈에 띔 (full-width, primary color)
- 정보 밀도: 높음 -- 날씨, 미세먼지, 체크리스트, 경로, 교통, 알림, 통계가 한 화면에 집중

**양호한 점:**
- Nielsen H1 (System Status): 날씨, 교통 실시간 정보 표시, 로딩 스피너 제공
- Gestalt 근접성: 관련 정보가 카드 단위로 그룹핑됨 (날씨 히어로, 출퇴근 카드, 통계 섹션)
- 날씨 체크리스트의 게이미피케이션: 준비물 체크/언체크가 `aria-pressed`로 접근성 지원
- 출퇴근 카드 내부의 정보 계층이 좋음: 경로 유형 배지 -> 이름 -> 상세 -> 교통 -> CTA
- 빈 상태 처리: 경로 없을 때 "출근 경로를 등록해보세요" + CTA

**발견된 이슈:**

| ID | 우선순위 | 휴리스틱/원칙 | 위치 | 현재 | 제안 | 근거 |
|----|---------|-------------|------|------|------|------|
| H-1 | P1 | Cognitive Load (Miller's Law) | 전체 | 한 화면에 7개 섹션: 인사, 날씨, 체크리스트, 출발예측, 경로추천, 출퇴근카드, 알림, 통계, 다른경로. 인지 부하 높음 | 날씨+체크리스트를 축약/접기 가능하게, 통계는 별도 탭/카드로 분리 | Miller's Law: 작업 기억은 7+-2개. 현재 9개 섹션은 초과 |
| H-2 | P1 | Nielsen H9 (Error Recovery) | 데이터 로드 실패 | `catch { }` -- 날씨/교통/통계 로드 실패 시 사용자에게 아무런 피드백 없음 | 로드 실패 시 해당 섹션에 "정보를 불러오지 못했어요" + 재시도 버튼 표시 | Silent failure. 사용자가 데이터 없음과 로드 실패를 구분 불가 |
| H-3 | P1 | Nielsen H4 (Consistency) | 스켈레톤 로딩 | `style={{ width: '160px', height: '24px' }}` 인라인 스타일 직접 지정 | CSS class로 스켈레톤 크기 관리. 실제 콘텐츠 레이아웃과 매칭되도록 | 스켈레톤이 실제 콘텐츠와 크기가 다르면 Layout Shift(CLS) 발생 |
| H-4 | P2 | Gestalt: Proximity | 알림 섹션 vs 통계 섹션 | 알림 CTA와 통계 섹션 사이 간격이 `.page { gap: 40px }`로 매우 큼 | 관련 섹션은 16px 간격, 비관련 섹션만 32px 간격으로 차등 적용 | 모든 섹션이 동일 40px 간격이면 그룹핑 인지가 약화됨 |
| H-5 | P2 | Fitts's Law | "출발하기" 버튼 | 버튼이 카드 내부 하단에 있어 스크롤 필요 | 스크롤 시에도 "출발하기" 버튼이 보이도록 sticky bottom CTA 고려 | 핵심 CTA가 스크롤 아래에 숨겨질 수 있음 |
| H-6 | P2 | Nielsen H6 (Recognition) | 경로 유형 토글 | "자동/출근/퇴근" 토글의 "자동"이 현재 무엇을 선택했는지 불분명 | "자동 (출근)" 또는 "자동 (퇴근)"처럼 현재 적용 결과를 표시 | 사용자가 "자동"을 눌렀을 때 어떤 경로가 선택되는지 회상해야 함 |
| H-7 | P3 | Information Architecture | 하드코딩된 좌표 | 서울 좌표(37.5665, 126.978) 고정 | 사용자 위치 API 또는 설정에서 좌표 관리 | 서울 외 사용자에게 잘못된 날씨 정보 |

### 2.3 경로 설정 페이지 (/routes)

**5초 테스트 (코드 기반):**
- 목록 화면: 헤더 + "새 경로" 버튼 + 탭 필터(전체/출근/퇴근) + 경로 카드 리스트
- 생성 플로우: 5단계 위저드 (유형 선택 -> 교통수단 -> 정류장 검색 -> 추가 질문 -> 확인)
- 빈 상태: SVG 아이콘 + "경로가 없어요" + CTA

**양호한 점:**
- Nielsen H3 (User Control): 뒤로가기 버튼이 모든 단계에 존재
- Nielsen H5 (Error Prevention): 경로 검증 훅(`useRouteValidation`)으로 최소 정류장, 중복 등 방지
- Gestalt: Common Region: 탭 필터가 경로 목록과 시각적 그룹
- 드래그 앤 드롭: `@dnd-kit/sortable`로 정류장 순서 변경 가능 (고급 사용자 지원)
- 수정 모드: 기존 경로 데이터가 폼에 프리로딩됨

**발견된 이슈:**

| ID | 우선순위 | 휴리스틱/원칙 | 위치 | 현재 | 제안 | 근거 |
|----|---------|-------------|------|------|------|------|
| R-1 | P1 | Nielsen H4 (Consistency) | 페이지 헤더 | 경로 페이지는 `apple-nav` (뒤로가기+제목), 홈/알림/설정은 `alert-page-v2-header` / `settings-page-v2-header` 스타일 | 모든 페이지 헤더를 통일된 패턴으로 | 3가지 다른 헤더 스타일이 혼재. 사용자가 "이것이 같은 앱인가" 의문 |
| R-2 | P2 | Nielsen H1 (System Status) | 경로 저장 시 | 저장 후 `navigate('/')` -- 홈으로 이동만 하고 성공 메시지 없음 | 저장 성공 시 토스트 메시지 표시 후 이동 | 사용자가 저장 성공 여부를 확인하려면 경로 목록을 다시 확인해야 함 |
| R-3 | P2 | Hick's Law | 탭 필터 | 경로가 1개뿐일 때도 "전체(1) / 출근(1) / 퇴근(0)" 3개 탭 표시 | 경로 2개 이하면 탭 숨기거나 축소 | 의미 없는 선택지가 인지 비용 추가 |
| R-4 | P2 | Nielsen H5 (Error Prevention) | 삭제 | 삭제 시 ConfirmModal 있음. 그러나 연결된 알림이 함께 삭제되는지 안내 없음 | "이 경로에 연결된 알림 N개도 영향을 받을 수 있습니다" 경고 추가 | 파괴적 행동의 부수효과를 숨기면 안 됨 |
| R-5 | P3 | Accessibility | 위저드 단계 간 전환 | 단계 변경 시 포커스가 새 섹션으로 이동하지 않음 | 단계 변경 시 새 섹션의 `<h1>` 또는 첫 번째 인터랙티브 요소로 포커스 이동 | 스크린리더 사용자가 내용 변경을 인지하기 어려움 |

### 2.4 알림 설정 페이지 (/alerts)

**5초 테스트 (코드 기반):**
- 기존 알림 목록 + "새 알림 추가" 버튼 + 위저드(5단계) + 빠른 프리셋
- 위저드 플로우: 유형 선택 -> 교통수단 -> 역 검색 -> 시간 설정 -> 확인

**양호한 점:**
- Nielsen H7 (Flexibility): "원클릭 설정" (빠른 날씨 알림)으로 파워유저와 초보자 모두 지원
- Nielsen H5 (Error Prevention): 중복 알림 감지 + 기존 수정/시간 변경 옵션 제공
- Gestalt: Similarity: `choice-card` 패턴으로 날씨/교통 선택이 일관됨
- `aria-pressed`, `aria-label`, `role="group"` 접근성 속성 적절

**발견된 이슈:**

| ID | 우선순위 | 휴리스틱/원칙 | 위치 | 현재 | 제안 | 근거 |
|----|---------|-------------|------|------|------|------|
| A-1 | P0 | Nielsen H1 (System Status) | 비로그인 상태 | Warning 배너만 표시 + 위저드 `display:none` 처리되지만 "단계" 텍스트와 체크마크가 노출됨 (스크린샷에서 확인) | `wizard-container`를 `!userId` 일 때 완전히 렌더링하지 않도록 조건부 렌더링 적용. `display:none` 대신 JSX 조건 분기 | 깨진 UI가 노출되어 앱 신뢰도 하락. CLAUDE.md 코드 품질 체크리스트 #1 위반 (조건부 렌더링 검증) |
| A-2 | P1 | Nielsen H4 (Consistency) | 비로그인 처리 | 경로/설정은 빈 상태 디자인(아이콘+텍스트+CTA), 알림은 warning 배너 + footer. 패턴 불일치 | 알림 페이지도 동일한 빈 상태 패턴 적용: 아이콘 + "로그인이 필요해요" + CTA | 같은 앱에서 비로그인 처리가 3가지 다른 패턴 |
| A-3 | P1 | Cognitive Load | 위저드 + 프리셋 + 기존 목록 | 한 페이지에 기존 알림 목록 + 위저드 + 빠른 프리셋이 동시 존재. "원클릭 설정"이 위저드 1단계와 페이지 하단 프리셋 두 곳에 중복 | 빠른 프리셋을 위저드 1단계에만 배치. 페이지 하단 `QuickPresets` 컴포넌트 제거 | 같은 기능이 2곳에 있으면 Hick's Law 위반 + 혼란 |
| A-4 | P2 | Peak-End Rule | 알림 생성 성공 | 성공 후 3초 뒤 자동으로 위저드가 리셋됨 | 성공 메시지 표시 시간을 5초로 늘리고, 사용자가 직접 닫기 가능하게 | 성공 경험(Peak moment)이 너무 빠르게 사라짐 |
| A-5 | P2 | Nielsen H6 (Recognition) | 알림 시간 표시 | Cron 표현식(`0 7 * * *`)을 파싱하여 "07:00"으로 표시. 요일 정보는 표시 안 됨 | "매일 07:00" 또는 "월~금 07:00"으로 사람 언어로 표현 | "07:00"만 보면 매일인지, 평일만인지 알 수 없음 |

### 2.5 설정 페이지 (/settings)

**5초 테스트 (코드 기반):**
- 4개 탭(프로필/경로/알림/앱) + 각 탭별 콘텐츠
- 아이콘 + 라벨 탭 패턴
- 뱃지로 경로/알림 개수 표시

**양호한 점:**
- Nielsen H6 (Recognition): 탭에 아이콘 + 라벨 + 뱃지로 상태 즉시 파악
- Nielsen H5 (Error Prevention): ConfirmModal로 삭제 확인, 이중 확인
- 개인정보 섹션: GDPR 스타일 데이터 내보내기/삭제 기능 제공 (좋은 실천)
- 버전 정보, 푸시 알림 토글 등 앱 설정이 체계적

**발견된 이슈:**

| ID | 우선순위 | 휴리스틱/원칙 | 위치 | 현재 | 제안 | 근거 |
|----|---------|-------------|------|------|------|------|
| S-1 | P1 | Nielsen H9 (Error Recovery) | 알림 토글/삭제 | `handleToggleAlert`: catch 블록이 빈 주석 ("Silent"). `handleDeleteConfirm`: catch 블록 빈 주석 | 실패 시 토스트로 "변경에 실패했습니다. 다시 시도해주세요." 표시. 토글 실패 시 UI 롤백 | 사용자가 토글했는데 서버에서 실패하면 UI와 서버 상태 불일치. Silent failure 금지 원칙 위반 |
| S-2 | P1 | Information Architecture | 경로/알림 탭 | 설정에서 경로 삭제/알림 토글이 가능하고, /routes와 /alerts에서도 동일 기능 제공 | 설정 > 경로/알림 탭을 "바로가기" 링크로 변경. 실제 관리는 전용 페이지에서만 | 기능 중복은 사용자를 혼란스럽게 함. "어디서 해야 하지?" |
| S-3 | P2 | Gestalt: Proximity | 앱 탭 | "앱 설정" -> "푸시 알림" -> "알림 발송 기록" -> "개인정보" 가 `style={{ marginTop: '1.5rem' }}` 인라인 스타일로 간격 조절 | CSS 클래스로 섹션 간격 관리. 인라인 스타일 제거 | 인라인 스타일은 일관성 파괴, 유지보수 저해 |

| S-5 | P3 | Nielsen H2 (Real World) | 사용자 ID | UUID 앞 4자리만 표시 ("xxxx..."). 복사 가능 | 사용자에게 의미 없는 ID 노출 축소. 닉네임/이름 우선 표시 | 개발자 관점 정보. 일반 사용자에게 불필요 |

## 3. 크로스-페이지 UX 패턴 분석

### 3.1 비로그인 처리 일관성 (P0 이슈)

현재 4가지 서로 다른 패턴이 사용됨:

| 페이지 | 패턴 | 문제 |
|--------|------|------|
| 홈 | 전용 GuestLanding (히어로+CTA+피쳐카드) | 적절 (메인 진입점) |
| 경로 | 빈 상태 디자인 (아이콘+메시지+CTA) in apple-nav | 적절 |
| 설정 | 빈 상태 디자인 (아이콘+메시지+CTA) in settings-header | 적절하나 경로와 스타일 차이 |
| **알림** | **warning 배너 + 깨진 위저드 텍스트** | **부적절. 깨진 UI 노출** |

**통일 제안:** 홈 이외의 모든 페이지에서 동일한 빈 상태 컴포넌트 사용:
```
<EmptyState
  icon={<LockIcon />}
  title="로그인이 필요해요"
  description="[기능명]을 사용하려면 로그인하세요"
  cta={{ label: "로그인", to: "/login" }}
/>
```

### 3.2 페이지 헤더 일관성 (P1 이슈)

| 페이지 | 헤더 클래스 | 스타일 |
|--------|-----------|--------|
| 홈 | `.home-header` | 좌측 인사말, 패딩 상이 |
| 경로 | `.apple-nav` | 좌측 뒤로가기 + 중앙 제목 + 우측 빈칸 |
| 알림 | `.alert-page-v2-header` | 좌측 정렬 "알림" |
| 설정 | `.settings-page-v2-header` | 좌측 정렬 "내 설정" |

홈은 고유하므로 예외, 나머지 3개 페이지의 헤더가 **3가지 다른 패턴**. 통일 필요.

### 3.3 빈 상태(Empty State) 패턴

| 페이지 | 빈 상태 | 평가 |
|--------|--------|------|
| 홈 (경로 없음) | "출근 경로를 등록해보세요" + CTA | 양호 |
| 경로 (목록 없음) | SVG 아이콘 + "경로가 없어요" + CTA | 양호 |
| 설정 > 경로 | "등록된 경로가 없어요" + CTA | 양호 |
| 설정 > 알림 | "설정된 알림이 없어요" + CTA | 양호 |
| 홈 (통계 없음) | "출퇴근 기록을 시작하면 통계를 볼 수 있어요" | 양호 |
| 알림 (없음) | 위저드가 자동 표시됨 | 양호 (빈 상태 대신 바로 액션) |

빈 상태 처리는 전반적으로 양호. "~없어요" 패턴이 일관됨.

### 3.4 피드백 패턴 (로딩/성공/에러)

| 패턴 | 구현 | 평가 |
|------|------|------|
| 페이지 로딩 | 스켈레톤 (HomePage), 스피너 (SettingsPage) | 불일치 |
| API 에러 | 대부분 Silent failure (`catch {}`) | **P1 문제** |
| 성공 피드백 | 알림 생성 시 toast, 설정 변경 시 toast | 양호 |
| 토글 피드백 | UI 즉시 반영 (optimistic) + 서버 실패 시 무시 | 롤백 없음 |

### 3.5 네비게이션 & 정보 구조

```
하단 네비: 홈 - 경로 - 알림 - 설정

문제:
1. "설정"에서 경로/알림 관리 가능 = 기능 중복
2. "알림 기록(/notifications)"은 설정 > 앱 > 링크로만 접근 = 숨겨진 경로
3. "/commute", "/commute/dashboard"는 하단 탭에 없음 = 발견 어려움
4. "/onboarding"은 첫 사용 후 접근 불가 = 일회성 (의도적이면 OK)
```

## 8. CSS 아키텍처 소견

### 현재 상태
- **16,873줄** 단일 CSS 파일 (`index.css`)
- CSS Custom Properties 기반 디자인 토큰 (양호)
- 41개 미디어 쿼리
- z-index 계층이 주석으로만 관리됨

### 핵심 문제
1. **단일 파일 관리 불가**: 특정 컴포넌트 스타일 변경 시 전체 파일을 검색해야 함
2. **미사용 CSS 누적**: 이전 버전의 스타일이 제거되지 않고 남아있을 가능성 높음
3. **클래스명 충돌 위험**: BEM 등 네이밍 규약 없이 flat한 클래스명 사용
4. **동일 요소 스타일 중복**: 같은 `.notice` 정의가 1806줄과 7910줄에 존재

### 단기 권고 (이번 사이클)
- CSS 파일 분리 자체는 L 노력이므로 이번 사이클에서는 하지 않음
- 대신 **새 컴포넌트/수정 시 인라인 스타일 금지** + **CSS 변수 적극 활용** 원칙 적용

### 장기 권고
- Tailwind CSS 점진적 도입 (글로벌 컨벤션)
- 또는 CSS Modules로 컴포넌트 단위 분리
- 미사용 CSS 감지 도구(PurgeCSS 등) 적용

## 9. 최종 판정

### NEEDS CHANGES

**P0 이슈: 1건** (알림 페이지 비로그인 깨진 UI)
**P1 이슈: 8건** (비로그인 패턴 불일치, Silent failure, 헤더 불일치, 색상 대비, 인지 부하, 기능 중복, ARIA)

### 잘된 점 요약
- 접근성 기본기가 뛰어남 (skip-link, ARIA, reduced motion, forced colors)
- 디자인 토큰 체계가 명확 (색상, 그림자, 모서리)
- 빈 상태 처리가 일관됨 ("~없어요" 패턴)
- 날씨 체크리스트의 게이미피케이션이 사용자 참여 유도에 효과적
- 하단 네비게이션의 프리패칭(touchStart/mouseEnter)이 성능에 좋음
- ConfirmModal로 파괴적 행동 이중 확인

### 가장 임팩트 있는 3가지 개선
1. **비로그인 처리 통일** -- 앱 전체 신뢰도 향상 (P0+P1, 노력 M)
2. **Silent failure 수정** -- 사용자 혼란 제거 (P1, 노력 M)
3. **`--ink-muted` 대비율 수정** -- WCAG AA 준수 (P1, 노력 S)
