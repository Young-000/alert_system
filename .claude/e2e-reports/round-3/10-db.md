## 4. RLS (Row Level Security)

| Table | RLS Enabled | Policies |
|-------|:-----------:|:--------:|
| `users` | NO | 0 |
| `alerts` | NO | 0 |
| `push_subscriptions` | NO | 0 |
| `subway_stations` | NO | 0 |
| `commute_routes` | NO | 0 |
| `route_checkpoints` | NO | 0 |
| `commute_sessions` | NO | 0 |
| `checkpoint_records` | NO | 0 |
| `weather_cache` | NO | 0 |
| `air_quality_cache` | NO | 0 |
| `subway_arrival_cache` | NO | 0 |
| `bus_arrival_cache` | NO | 0 |
| `api_call_log` | NO | 0 |

> **All 13 tables have RLS disabled and 0 policies.** The frontend Supabase client uses `alert_system` schema with anon key. Without RLS, any client with the anon key can read/write all data in these tables. This is mitigated by:
> 1. The backend uses direct DB connection (TypeORM), not PostgREST.
> 2. The frontend Supabase client is only used for the `users` table (auth).
> 3. Cache tables are non-sensitive.
>
> However, `users`, `alerts`, `push_subscriptions` contain user data and should have RLS.
