## 3. 타임스탬프 (`created_at`, `updated_at`)

## 4. Boolean 컬럼 네이밍 (`is_` / `has_` 접두사)

| 파일 | 컬럼 | 상태 | 비고 |
|------|------|------|------|
| `alert.entity.ts` | `enabled: boolean` | ⚠️ 기술 부채 | 도메인 레이어 전체에 `Alert.enabled` 으로 deeply embedded. 자동 수정 시 domain entity, DTOs, services, tests 전체 수정 필요. `isEnabled`로의 rename은 별도 리팩토링으로 처리 권장 |
| `notification-rule.entity.ts` | `enabled: boolean` | ⚠️ 기술 부채 | 동일 이유. 단, `isSystemRule: boolean`은 올바른 패턴으로 혼재 상태 |

## 8. Repository 패턴

### Raw Query 사용
- `createQueryBuilder` 사용 시 모두 파라미터 바인딩 (`{ userId }`, `:beforeDate`) 사용 ✅
- `queryRunner.query()` (seed 파일): `$1` 파라미터 방식 사용 ✅
- 문자열 직접 연결에 의한 SQL Injection 위험 없음 ✅

### N+1 패턴
- 모든 repository에서 `find()` / `findOne()`을 개별 호출로 사용하나, 루프 내 DB 호출 패턴은 발견되지 않음 ✅
- `map()` 사용은 메모리 내 변환(toDomain)으로 DB 쿼리 없음 ✅
