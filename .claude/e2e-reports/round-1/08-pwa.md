  - `WebPushService` at `backend/src/infrastructure/messaging/web-push.service.ts`:

    - `sendToUser()` sends to all user's web subscriptions

## 8-4: EventBridge Scheduler

**Status**: PASS

- `EventBridgeSchedulerService` at `backend/src/infrastructure/scheduler/eventbridge-scheduler.service.ts`:
  - Implements `INotificationScheduler` interface
  - Uses `@aws-sdk/client-scheduler` with `SchedulerClient`
  - `scheduleNotification()`: Creates or updates recurring schedules
  - `cancelNotification()`: Deletes schedules, ignores `ResourceNotFoundException`
  - Cron conversion: standard 5-field -> EventBridge 6-field (handles ranges, lists, `*`, HH:mm format)
  - Schedule Group: configurable via `SCHEDULE_GROUP_NAME` (default: `alert-system-prod-alerts`)
  - Timezone: `Asia/Seoul`
  - Retry: 3 attempts, 1-hour max age
  - DLQ support when configured
  - Proper error propagation for non-ResourceNotFound errors
- `SchedulerModule` provides DI via `INotificationScheduler` token
- `SchedulerTriggerController` at `backend/src/presentation/controllers/scheduler-trigger.controller.ts`:
  - `POST /scheduler/trigger` -- receives EventBridge callbacks
  - `POST /scheduler/weekly-report` -- weekly report trigger (B-6)
  - `POST /scheduler/health` -- health check for EventBridge target validation
  - All endpoints use `timingSafeEqual` for `x-scheduler-secret` header verification
  - `@Public()` decorator bypasses JWT auth (uses scheduler secret instead)
- `ScheduleDepartureAlertsUseCase` for smart departure one-time schedules:
  - Uses `at()` expressions for one-time triggers
  - `ActionAfterCompletion.DELETE` for auto-cleanup
  - Supports pre-alert scheduling (30min, 10min, 0min before departure)
- Comprehensive tests: `eventbridge-scheduler.service.spec.ts` (cron conversion, create/update/delete, error handling)

## 8-5: Solapi Alimtalk

**Status**: PASS

- `SolapiService` at `backend/src/infrastructure/messaging/solapi.service.ts`:
  - Uses `solapi` npm package (`SolapiMessageService`)
  - All config from env: `SOLAPI_API_KEY`, `SOLAPI_API_SECRET`, `SOLAPI_PF_ID`, `SOLAPI_TEMPLATE_ID`
  - No hardcoded sensitive values (API key/secret from ConfigService)
  - Template IDs defined as `TEMPLATE_IDS` constant object:
    - Morning/evening variants for weather, transit, combined alerts
    - Legacy template `KA01TP2601181035243285qjwlwSLm5X` for backward compatibility
  - Methods: `sendWeatherAlert`, `sendTransitAlert`, `sendCombinedAlert`, `sendLegacyWeatherAlert`, `sendWeeklyReport`
  - Phone number formatting (hyphen removal)
  - Graceful degradation: `NoopSolapiService` when credentials not configured
  - Mock mode: logs instead of sending when unconfigured
  - Error handling: logs and re-throws on send failure
- `MessagingModule` factory decides between `SolapiService` and `NoopSolapiService` based on env
- `NotificationModule` also provides `SOLAPI_SERVICE` via factory with same logic
- `AlimtalkService` (NHN Cloud) exists as legacy alternative at `backend/src/infrastructure/messaging/alimtalk.service.ts`
  - Timeout handling with `AbortController`
  - SMS fallback via `resendParameter`
- `SendNotificationUseCase` orchestrates: data collection -> message building -> Solapi send -> web push -> notification log
  - Fallback: if new template fails, tries legacy template
  - Logs all send results (success/fallback/failed)
- Comprehensive tests: `solapi.service.spec.ts` covers all alert types, error cases, noop service
