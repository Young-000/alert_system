## 8. Fixes Applied

### Fix 1: NotificationRuleEntity missing onDelete (FIXED)

**File**: `backend/src/infrastructure/persistence/typeorm/notification-rule.entity.ts`

**Problem**: `@ManyToOne(() => UserEntity, { nullable: true })` had no `onDelete` specified. If a user is deleted, notification rules would become orphaned with a dangling FK reference, causing constraint violations.

**Fix**: Added `onDelete: 'SET NULL'` since the relation is nullable (system rules have no user).

```typescript
// Before
@ManyToOne(() => UserEntity, { nullable: true })

// After
@ManyToOne(() => UserEntity, { nullable: true, onDelete: 'SET NULL' })
```

### Fix 2: Cache entities using `timestamp` instead of `timestamptz` (FIXED)

**Files**:
- `backend/src/infrastructure/persistence/typeorm/weather-cache.entity.ts`
- `backend/src/infrastructure/persistence/typeorm/air-quality-cache.entity.ts`
- `backend/src/infrastructure/persistence/typeorm/transport-cache.entity.ts` (2 occurrences)

**Problem**: 4 cache entities used `type: 'timestamp'` (timezone-unaware) for `expires_at` columns, while all other timestamp columns in the project use `timestamptz`. In the Korean timezone (UTC+9), comparing `timestamp` with `timestamptz` can cause 9-hour offset bugs in cache expiration logic.

**Fix**: Changed all 4 occurrences from `timestamp` to `timestamptz`.

```typescript
// Before (4 occurrences across 3 files)
@Column({ name: 'expires_at', type: 'timestamp' })

// After
@Column({ name: 'expires_at', type: 'timestamptz' })
```

### Fix 3: DatabaseModule missing 6 entities in forFeature (FIXED)

**File**: `backend/src/infrastructure/persistence/database.module.ts`

**Problem**: The global `DatabaseModule` was missing 6 newer entities from its `TypeOrmModule.forFeature()` registration:
- `MissionEntity`, `DailyMissionRecordEntity`, `MissionScoreEntity`
- `CommuteStreakOrmEntity`, `StreakDailyLogOrmEntity`
- `LiveActivityTokenEntity`

While `autoLoadEntities: true` covers these via feature modules, the global module should maintain a complete registry for consistency and to ensure repositories can inject these entities from any module.

**Fix**: Added all 6 entities with proper imports.

## 9. Observations (No Fix Needed)

### 9-1. StreakDailyLogOrmEntity has sessionId without ManyToOne relation

`StreakDailyLogOrmEntity` has a `sessionId` column but no `@ManyToOne` to `CommuteSessionEntity`. This appears to be an intentional loose coupling -- the streak log references a session ID but doesn't create a hard FK dependency. This is acceptable since:
- The streak system doesn't need to eagerly load session data
- Session deletion should not cascade-delete streak logs

### 9-2. UserEntity has no OneToMany back-references

The `UserEntity` defines no `@OneToMany` arrays. This is a deliberate performance choice -- loading a user never triggers cascading loads of alerts, routes, sessions, etc. All queries go through dedicated repositories.

### 9-3. Frontend-Backend Type Alignment

Frontend API client types (`mission-api.client.ts`, `commute-api.client.ts`) align well with backend entity structures. The API response types correctly mirror entity fields through the controller/use-case mapping layer.
