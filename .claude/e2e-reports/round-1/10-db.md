## 7. 쿼리 최적화 (Relations 과도 로드)

`commute-session.repository.ts`의 목록 조회 대부분이 기본으로 `relations: ['checkpointRecords']` 포함:

```typescript
async findByUserId(userId: string, limit = 50): Promise<CommuteSession[]> {
  const entities = await this.sessionRepository.find({
    where: { userId },
    relations: ['checkpointRecords'],  // 목록에서도 체크포인트 로드
    order: { startedAt: 'DESC' },
    take: limit,
  });
}
```

50개 세션 × N개 체크포인트 = 과도한 데이터 로드 가능. 사용 컨텍스트에 따라 목록 조회 시 relations 제거 고려. 현재 기능적 문제는 없어 수정 보류.

**결론**: WARN (수정 보류)

## 8. Fixes Applied

### Fix 1: notification_logs.alertId FK 누락 (FIXED)

**File**: `backend/src/infrastructure/persistence/typeorm/notification-log.entity.ts`

**Problem**: `alertId` 컬럼이 `alerts` 테이블을 참조하는 UUID를 저장하지만 FK 선언 없음. 알림 삭제 시 notification_log 레코드가 dangling UUID를 갖는 orphaned record 상태가 됨.

**Fix**:
1. `alertId`를 `nullable: true`로 변경 (`string | null` 타입)
2. `AlertEntity`에 대한 `@ManyToOne` FK 추가 (`onDelete: 'SET NULL'`)
3. `alertId` 인덱스 추가

```typescript
// Before
@Column({ type: 'uuid', name: 'alert_id' })
alertId: string;  // FK 선언 없음

// After
@Column({ type: 'uuid', name: 'alert_id', nullable: true })
alertId: string | null;

@ManyToOne(() => AlertEntity, { nullable: true, onDelete: 'SET NULL' })
@JoinColumn({ name: 'alert_id' })
alert?: AlertEntity;
```

### Fix 2: commute-route update() 트랜잭션 누락 (FIXED)

**File**: `backend/src/infrastructure/persistence/repositories/commute-route.repository.ts`

**Problem**: `update()` 메서드에서 체크포인트 삭제(`DELETE`) 후 새 체크포인트 저장(`INSERT`)이 트랜잭션 없이 수행됨. 체크포인트 저장 실패 시 기존 체크포인트가 삭제된 상태로 남는 데이터 불일치 발생.

**Fix**: `DataSource.transaction()` 적용 — 라우트 저장 + 체크포인트 삭제 + 체크포인트 저장을 원자적 트랜잭션으로 처리.

```typescript
// Before
async update(route: CommuteRoute): Promise<void> {
  await this.routeRepository.save(entity);
  await this.checkpointRepository.delete({ routeId: route.id });  // 삭제만 됨
  await this.checkpointRepository.save(checkpointEntities);        // 실패 가능
}

// After
async update(route: CommuteRoute): Promise<void> {
  await this.dataSource.transaction(async (manager) => {
    await manager.save(CommuteRouteEntity, entity);
    await manager.delete(RouteCheckpointEntity, { routeId: route.id });
    if (route.checkpoints.length > 0) {
      await manager.save(RouteCheckpointEntity, checkpointEntities);
    }
  });
}
```

### Fix 3: commute-session update() 트랜잭션 누락 (FIXED)

**File**: `backend/src/infrastructure/persistence/repositories/commute-session.repository.ts`

**Problem**: `update()` 메서드에서 세션 저장과 체크포인트 레코드 저장이 트랜잭션 없이 수행됨.

**Fix**: `DataSource.transaction()` 적용.

## 9. Observations (수정 불필요)

### 9-1. Soft Delete 미사용

`@DeleteDateColumn` 미사용. 전체 entity가 hard delete 방식. 프로젝트 초기 단계로 현재 요구사항 범위 내. 향후 `users`, `alerts`, `commute_routes`에 우선 적용 권장.

### 9-2. StreakDailyLogOrmEntity.sessionId FK 없음

`sessionId` 컬럼이 있지만 `@ManyToOne` 없음 — 의도적 느슨한 결합. 세션 삭제 시 streak log가 삭제되지 않도록 하기 위한 설계.

### 9-3. Migration 파일 없음

TypeORM `synchronize` 의존 방식. migration 파일이 없으나 `buildDataSourceOptions()`에 entity 목록이 완전히 관리되고 있음.
